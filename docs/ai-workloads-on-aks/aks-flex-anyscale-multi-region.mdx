## Objective

AI teams often have a placement problem before they have a modeling problem. The
workload is ready, but the GPU capacity may be in a different Azure region, in a
lab datacenter, or in another cloud environment. AKS Flex Node gives AKS a way to use
that Linux compute as worker capacity. Anyscale on Azure gives the team a managed
Ray control plane for Jobs, compute profiles, and workload observability. Together,
they let you run Ray AI/ML workloads where your compute and GPUs already are.

By the end, you will have one AKS-backed Anyscale cloud with a Flex host joined
from another region. The important part is the proof: the workload must show that
the Ray worker ran on the Flex node, and the teardown gate must show that the
current lab resources are gone. The CPU path gives you the cheapest repeatable
run. If you have T4 quota in the Flex region, the GPU path proves a `GPU:1`
worker landed on a Flex-hosted Tesla T4.

## Topology flexibility

This lab uses **two Azure regions**: an AKS cluster homed in `westus2` with Flex
hosts joined from `westus3` for the CPU path, or `southcentralus` in the validated
T4 GPU run. That is one supported Flex Node shape, not the only one.

:::info What AKS Flex Node actually allows
According to the [AKS Flex Node documentation](https://github.com/Azure/AKSFlexNode), any Linux host that can reach the AKS API server over outbound HTTPS can join as a worker node. In this lab that host is an Azure VM in a different region, but the same pattern can apply to on-premises bare metal, on-premises VMs, or VMs in another cloud environment when the network path is in place.

The host does not need to be in Azure. The key requirement is network reachability to the AKS API server and bidirectional pod-to-pod routing between the Flex host and AKS nodes. Swap `westus3` in this lab for your on-premises network range and the steps are identical.
:::

:::warning GPU support status
GPU Flex Node support is listed as **"under active validation"** (alpha) in the AKS Flex Node repository.
AKS Flex Node does **not** install the NVIDIA kernel driver. The host image must have the driver pre-baked.
For GPU runs, use a Linux image with the matching NVIDIA driver already installed.
The CPU path has no such constraint.
:::

## Choose your path

This lab is built so you can complete it end to end on either path. **The CPU path is the primary route** and produces cross-region placement proof without GPU quota. The GPU path is an optional extension for saturation evidence after quota and driver readiness are confirmed.

| Dimension | CPU path | GPU path |
| --- | --- | --- |
| Flex host SKU | Any 4+ vCPU Linux VM | Standard_NC16as_T4_v3 or equivalent |
| Quota needed | Standard DSv5 family in both regions | Standard NCASv3_T4 family in the Flex region |
| Driver required | No | Yes, pre-baked in the host image |
| Workload | `train.py --cpu-only` | `train.py` (GPU default) |
| Evidence | Placement and scaling proof | Placement, GPU saturation, and scaling proof |

## Prerequisites

### Common (both paths)

- An Azure subscription with permission to create AKS, networking, storage, and monitoring resources
- Azure CLI installed and authenticated
- Terraform 1.9 or later
- Basic familiarity with AKS, Kubernetes, and managed identities

### GPU path only

- GPU quota available in the **Flex host region**:
  - `Standard_NC16as_T4_v3` = 16 vCPU per node
  - Minimum: 16 vCPU in the Flex region (1 T4 Flex host)
  - Larger runs require quota for each additional Flex GPU host
- Host image with NVIDIA driver pre-installed. This lab uses `microsoft-dsvm/ubuntu-hpc/2204`.

:::tip Checking your quota

```bash
az vm list-usage --location westus3 --query "[?name.value=='standardNCASv3_T4Family']" -o table
az vm list-usage --location southcentralus --query "[?name.value=='standardNCASv3_T4Family']" -o table
```

If your intended Flex region shows 0 available, use the CPU path, choose another validated Flex region, or request a quota increase in [Azure Quotas](https://portal.azure.com/#view/Microsoft_Azure_Capacity/QuotaMenuBlade).
:::

## Architecture

The AKS cluster is homed in `westus2` and owns the Kubernetes control plane, Anyscale cloud binding, and home-region node pools. Flex hosts in the second region join the same cluster over VNet peering and appear as standard Kubernetes worker nodes. Anyscale sees one cluster and schedules Ray workers onto nodes that match the compute profile.
