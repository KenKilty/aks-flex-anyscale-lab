Before running the workload proof, you need to confirm that the CPU scale profile,
Flex service routing, Flex DNS, and Anyscale Gateway API path are healthy. This
module is the final preflight for Gate 5; it does not run the proof workload.

## Objectives

By the end of this module you will be able to:

- Set quota-safe autoscaling bounds for the CPU node pool
- Understand the difference between home-region AKS node pool autoscaling and Flex host capacity
- Re-run the explicit Flex Ready, kube-proxy-flex, DNS, and Anyscale health checks
- Confirm the cluster is ready for the CPU proof, or for the GPU proof when T4 quota is enabled

<Prerequisites />

## Understanding the capacity model

This lab has two distinct compute locations:

| Location | Managed by | Scales how |
| ---------- | ----------- | ------------ |
| Home-region AKS CPU pool (`cpu`) | AKS cluster autoscaler | Adds nodes from Azure capacity on demand |
| Flex host in `westus3` | Pre-provisioned in Module 3 | Fixed capacity — always `Ready` |

The Flex host does **not** autoscale in this lab. It is pre-provisioned and permanently
joined. Anyscale can place workers on it immediately without waiting for a scale event.

## Step 1 — Review current pool configuration

```bash
az aks nodepool list \
  --resource-group rg-flexany-dev-wus2 \
  --cluster-name aks-flexany-dev-wus2 \
  -o table
```

Expected output:

```text
Name    OsType    VmSize                 Count    MinCount    MaxCount    Mode
------  --------  ---------------------  -------  ----------  ----------  ------
sys     Linux     Standard_D4s_v5         1        1           3           System
cpu     Linux     Standard_D4s_v5         0        0           4           User
```

## Step 2 — Set quota-aware bounds in `.env`

### CPU path

```bash
TF_VAR_cpu_pool_min_count="0"
TF_VAR_cpu_pool_max_count="4"
TF_VAR_gpu_pool_configs='{}'
```

Expected output after saving `.env` and checking the values:

```text
TF_VAR_cpu_pool_min_count="0"
TF_VAR_cpu_pool_max_count="4"
TF_VAR_gpu_pool_configs='{}'
```

### GPU path

Choose the GPU path in Module 3 before provisioning the Flex host. If you are
continuing from a CPU-only host, changing the Flex VM size or image will replace
that host; use a fresh lab run for the cleanest student path.

```bash
TF_VAR_flex_host_vm_size="Standard_NC16as_T4_v3"
TF_VAR_flex_host_source_image_reference='{"publisher":"microsoft-dsvm","offer":"ubuntu-hpc","sku":"2204","version":"latest"}'
ANYSCALE_FLEX_GPU_ENABLED="true"
ANYSCALE_PROOF_GPU_ACCELERATOR_TYPE="T4"
ANYSCALE_PROOF_GPU_PRODUCT_LABEL="NVIDIA-T4"
ANYSCALE_PROOF_GPU_TARGET="flex"
ANYSCALE_PROOF_GPU_WORKER_COUNT="1"
```

Only enable this after the CPU proof succeeds and you have confirmed T4 quota in
the Flex region. GPU workers in this lab must run on the Flex node, not an AKS
managed GPU pool.

## Step 3 — Apply the profile

```bash
./scripts/anyscale-aks.sh apply
```

This updates the node pool `min_count`/`max_count` without replacing nodes.
If you changed the Flex host VM size or image for the GPU path, Terraform will
replace the Flex host so the new GPU-capable image can boot.

Expected output:

```text
Apply complete! Resources: 0 added, 1 changed, 0 destroyed.
```

## Step 4 — Run the final preflight gates

If you enabled the GPU path, first confirm the Flex host driver and then install
the NVIDIA device plugin before running the gate. The helper pins the plugin to
the Flex node and adds the taint tolerations used by this lab.

```bash
ssh -i "$SSH_PRIVATE_KEY_PATH" \
  "${TF_VAR_flex_host_admin_username}@$(./scripts/anyscale-aks.sh output -raw flex_host_public_ip)" \
  'nvidia-smi --query-gpu=name,driver_version --format=csv,noheader'
```

Expected output:

```text
Tesla T4, <driver-version>
```

The helper pins the plugin to the Flex node and labels that node with
`nvidia.com/gpu.product=NVIDIA-T4`, matching the selector Anyscale adds for T4
GPU workers.

```bash
./scripts/install-nvidia-device-plugin.sh
```

Expected output:

```text
daemonset.apps/nvidia-device-plugin-daemonset configured
daemon set "nvidia-device-plugin-daemonset" successfully rolled out
NVIDIA device plugin ready: 1 allocatable GPU(s) in pool aksflexnodes
```

Use the gate script instead of pasting long diagnostic pods by hand. It checks the
CPU scale profile, optional GPU pool setting, Flex node readiness, Flex service
routing, Flex DNS, Flex HTTPS egress, AKS-to-Flex pod reachability, and the
Anyscale extension/Gateway path.

```bash
./scripts/validate-lab-gates.sh m5
```

Expected output:

```text
PASS M5-01 CPU node pool autoscaling configured
PASS M5-02 Flex GPU node available with allocatable GPU
PASS Flex node vm-flex-flexany-dev-wus3 Ready in westus3
PASS kube-proxy-flex ready 1/1
PASS Flex ClusterFirst DNS resolves Anyscale and Kubernetes service names
PASS Flex pod HTTPS egress reaches console.azure.anyscale.com
PASS AKS pod reached Flex pod 10.244.0.36:8080
PASS Anyscale extension and operator pods ready
PASS Gateway anyscale-operator/anyscale-gateway programmed at 20.190.18.76
```

The helper writes raw pod logs and Kubernetes JSON under
`.github/agents/state/m5-flex-network/`. If a check fails, use that directory first;
it contains the short-lived diagnostic pod output before cleanup.

## Step 5 — Inspect Anyscale extension and Gateway health

The gate script already checked this. Use these commands when you want the raw
extension and Gateway details.

```bash
az k8s-extension show \
  --cluster-name aks-flexany-dev-wus2 \
  --resource-group rg-flexany-dev-wus2 \
  --cluster-type managedClusters \
  --name anyscale-operator \
  --query '{name:name,provisioningState:provisioningState,currentVersion:currentVersion}' -o json

kubectl get pods,gateway -n anyscale-operator -o wide
```

Expected output:

```text
{
  "name": "anyscale-operator",
  "provisioningState": "Succeeded",
  "currentVersion": "1.7.1"
}

pod/anyscale-operator-556c4df7cb-tq4r7   3/3   Running
gateway.gateway.networking.k8s.io/anyscale-gateway   approuting-istio   20.190.18.76   True
```

:::warning Stop before Gate 5
Do not start proof loops if any check above fails. Gate 5 assumes Flex pod IP
capacity, Flex-only service routing, ClusterFirst DNS, and Anyscale Gateway API are
already healthy.
:::

## Step 6 — Confirm no stale proof pods are active

```bash
kubectl get pods -A -o wide --field-selector spec.nodeName="$FLEX_NODE"
```

Expected output:

```text
NAMESPACE     NAME                    READY   STATUS    NODE
kube-system   kube-proxy-flex-abcde   1/1     Running   vm-flex-flexany-dev-wus3
```

## Scale profile summary

Before proceeding, confirm your scale profile matches your quota:

```bash
./scripts/anyscale-aks.sh status | jq '.flex_join_contract, .foundation_contract.aks_contract'
```

Expected output:

```json
{
  "enabled": true,
  "agent_pool_name": "aksflexnodes",
  "vm_name": "vm-flex-flexany-dev-scus"
}
{
  "public_aks": true
}
```

The exact Flex VM suffix follows `TF_VAR_flex_region_short`; CPU examples commonly
use `wus3`, while the validated T4 GPU path uses `scus`.
