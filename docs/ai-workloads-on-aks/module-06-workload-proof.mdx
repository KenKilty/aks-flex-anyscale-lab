This module runs the `deepspeed_finetune` proof workload — a Ray Train + DeepSpeed
distributed training script adapted for this lab. The organized path is CPU-only:
the Ray head lands on AKS capacity in `westus2`, and the worker lands on the Flex
node in `westus3`. Each worker emits its hostname, region hint, and device details
as structured job output. After the run, the proof harness extracts and validates
that output so you can prove the workload executed on nodes in both locations.

## Objectives

By the end of this module you will be able to:

- Submit the CPU-only proof through Anyscale after Gates 1 through 4 pass
- Confirm the head pod lands on AKS CPU capacity and the worker pod lands on Flex capacity
- Confirm per-worker region placement from machine-readable output
- Validate the proof summary against the expected schema

## Path differences

| Step | CPU path | GPU path |
| ------ | ---------- | ------------ |
| Compute config | Created by `./scripts/run-anyscale-proof.sh --mode cpu` | Created by `./scripts/run-anyscale-proof.sh --mode gpu` |
| Workers | CPU-only Ray actors on the Flex node | GPU-accelerated Ray Train workers on the AKS GPU pool |
| Evidence | Placement + loss curve + log-emitted proof summary | Placement + GPU utilisation + loss curve |
| Duration | ~10 min | ~20–40 min (includes autoscale wait) |

## Step 1 — Confirm Gate 5 can start

Run the proof only after the Module 5 preflight checks pass. The Flex diagnostic pod
must use normal `ClusterFirst` DNS and the Flex node must not carry the broad
`kubernetes.azure.com/cluster` label.

```bash
source .env
FLEX_NODE="vm-flex-${TF_VAR_project}-${TF_VAR_environment}-${TF_VAR_flex_region_short}"
kubectl get node "$FLEX_NODE" --show-labels | grep -v 'kubernetes.azure.com/cluster'
kubectl -n kube-system rollout status ds/kube-proxy-flex --timeout=3m
```

Expected output:

```text
vm-flex-flexany-dev-wus3   Ready   agent   22m   v1.31.1   agentpool=aksflexnodes,...,topology.kubernetes.io/region=westus3
daemon set "kube-proxy-flex" successfully rolled out
```

## Step 2 — Choose a run profile

Set `profile` in the workload call:

- `smoke` — 1 epoch, 4 debug steps, 2 workers (fast validation, ~10 min)
- `full` — all epochs, full dataset, up to 4 workers (full saturation, ~40 min GPU)

For a first run, start with `smoke`.

## Step 3 — CPU path — local smoke run

Validate the workload logic without a cluster:

```bash
./scripts/run-workload-smoke.sh
```

This runs `train.py --profile smoke --cpu-only` locally and produces a proof
summary at `.cache/workloads/deepspeed_finetune/smoke/proof-summary.json`.

Validate the output:

```bash
python3 workloads/deepspeed_finetune/validate_proof_summary.py \
  .cache/workloads/deepspeed_finetune/smoke/proof-summary.json
```

Expected output:

```text
validated: .cache/workloads/deepspeed_finetune/smoke/proof-summary.json
```

The proof summary schema includes:

- `run_id`, `experiment_name`, `timestamp_unix`, and profile fields
- `placement`: expected regions and worker placement hints
- `metrics`: loss curve and runtime counters
- `worker_snapshot`: per-worker hostname, region_hint, and device
- `storage_proof`: managed-identity storage account, container, path, and fsspec/adlfs `az://` URI

## Step 4 — Submit through Anyscale (CPU path)

```bash
./scripts/run-anyscale-proof.sh --mode cpu
```

Expected output:

```text
Gateway anyscale-operator/anyscale-gateway programmed address: 172.193.180.193
(anyscale +1.2s) Created compute config: 'cpu-home:6'
info: submit attempt 1/3 for flex-proof-cpu-20260708-235149
(anyscale +2m26.1s) Job 'flex-proof-cpu-20260708-235149' transitioned from STARTING to RUNNING
(anyscale +5m1.0s) Job 'flex-proof-cpu-20260708-235149' reached target state, exiting
```

:::note Gate 5 placement contract
The CPU proof is the primary workshop proof. It must place the Ray head on AKS CPU
capacity in `westus2`, place the Ray worker on `vm-flex-flexany-dev-wus3`, and keep
worker DNS policy as `ClusterFirst`.
:::

## GPU path — submit through Anyscale

Run this only after Module 5 reports `PASS M5-02 GPU node pool available with
allocatable GPU`. The GPU proof uses the AKS GPU node pool declared in
`TF_VAR_gpu_pool_configs`; it does not place GPU workers on the Flex host unless you
provide a Flex image with NVIDIA drivers and runtime support.

```bash
./scripts/run-anyscale-proof.sh --mode gpu
```

Expected output includes GPU compute config creation and an autoscale wait if the
second T4 worker needs another node:

```text
(anyscale +1.3s) Created compute config: 'gpu-home:1'
info: submit attempt 1/3 for flex-proof-gpu-20260709-110159
(anyscale +6m44.7s) Job 'flex-proof-gpu-20260709-110159' transitioned from STARTING to RUNNING
```

The first GPU run can spend several minutes scaling a second T4 node and pulling the
CUDA Ray image. The proof harness gives GPU workers a longer startup window than the
CPU path for that reason.

## Step 6 — Retrieve and validate the proof summary

For local smoke runs, the summary is written locally under `.cache/workloads/...`.
For Anyscale job runs, `./scripts/run-anyscale-proof.sh` extracts the
`PROOF_SUMMARY_JSON` log record and writes it under `.cache/anyscale/proofs/`.
It also writes Kubernetes placement evidence for the job, including pod IP, node,
node region, node pool, and Ray node type.
The workload also writes and reads the same summary through the Anyscale workload
managed identity using the fsspec/adlfs `az://` protocol; no SAS token or shared
storage key is used.

:::note Storage RBAC
The Anyscale workload managed identity needs `Storage Blob Data Contributor` on the
default container because the proof writes and reads evidence. A local operator who
wants to download that blob with `az storage ... --auth-mode login` also needs a
data-plane role such as `Storage Blob Data Reader` or `Storage Blob Data Contributor`.
Management-plane ownership alone is not enough for blob data reads.
:::

```bash
OUT=.cache/anyscale/proofs/<proof-name>-proof-summary.json

python3 workloads/deepspeed_finetune/validate_proof_summary.py \
  "$OUT"
```

Expected output:

```text
validated: .cache/anyscale/proofs/flex-proof-cpu-20260708-1712-proof-summary.json
```

Expected proof-summary shape:

```json
{
  "run_id": "flex-proof-cpu-20260708-192330",
  "profile": "smoke",
  "placement": {
    "expected_regions": ["westus2", "westus3"],
    "observed_region_hint": "westus3",
    "observed_node_hint": "k-be9e65e2681b30000",
    "observed_world_size": 1
  },
  "storage_proof": {
    "account_name": "stflexanydevwus2",
    "container_name": "flexany-dev-blob",
    "client_id_present": true,
    "path": "flexany-dev-blob/anyscale/proofs/<proof-name>/proof-summary.json",
    "uri": "az://flexany-dev-blob/anyscale/proofs/<proof-name>/proof-summary.json"
  }
}
```

## Step 7 — Verify placement

Open the proof summary and check the `placement` and `storage_proof` sections:

```bash
jq '{placement, storage_proof}' .cache/anyscale/proofs/<proof-name>-proof-summary.json
jq '.pods[] | {name, pod_ip, node_name, node_region, node_agentpool, ray_node_type}' \
  .cache/anyscale/proofs/<proof-name>-kubernetes-placement.json
```

Expected output:

```json
{
  "placement": {
    "expected_regions": ["westus2", "westus3"],
    "observed_world_size": 1
  },
  "storage_proof": {
    "client_id_present": true,
    "uri": "az://..."
  }
}
{
  "name": "k-flex-proof-cpu-worker-group-0-fghij",
  "pod_ip": "10.244.0.36",
  "node_name": "vm-flex-flexany-dev-wus3",
  "node_region": "westus3",
  "node_agentpool": "aksflexnodes",
  "ray_node_type": "worker"
}
```

Derive the expected region values from `.env` before judging pass/fail:

```bash
source .env
echo "home=${TF_VAR_azure_location} flex=${TF_VAR_flex_region}"
```

Verify the worker ran on Flex by checking worker log lines for the Ray worker IP,
then confirm Kubernetes placed that worker pod on the Flex node. In this lab, the
worker pod IP comes from `TF_VAR_flex_pod_cidr` (for example `10.244.0.0/16`), and
AKS reaches it through the Module 3 route table rather than by treating it as an AKS
subnet IP.

```bash
.venv/bin/anyscale job logs --name <job-name> | grep 'RayTrainWorker pid='

kubectl get pods -n anyscale-operator \
  -l "app.kubernetes.io/name=<job-name>" \
  -o wide --show-labels
```

Expected output:

```text
RayTrainWorker pid=336, ip=10.244.0.36
k-be9e65e2681b30000   5/5   Running   10.244.0.36   vm-flex-flexany-dev-wus3   ... ray-node-type=worker
```

The proof validator also checks that the remote CPU worker summary reports the Flex
region, and the harness checks that a Kubernetes worker pod lands in the Flex
region. A stale or globally injected home-region hint should fail validation rather
than being accepted as placement evidence.

## Flex GPU host note

The GPU proof above validates AKS GPU capacity. A GPU-sized Flex VM can join the
cluster, but the default Ubuntu image does not include NVIDIA host libraries or the
container runtime integration required to advertise `nvidia.com/gpu`. To run GPU
workers on Flex, use a pre-baked GPU image and verify the host first:

```bash
ssh -i ~/.ssh/id_ed25519 azureoperator@$(./scripts/anyscale-aks.sh output -raw flex_host_public_ip) \
  'nvidia-smi --query-gpu=utilization.gpu,memory.used --format=csv,noheader,nounits'
```

A value above 80% confirms GPU saturation.
