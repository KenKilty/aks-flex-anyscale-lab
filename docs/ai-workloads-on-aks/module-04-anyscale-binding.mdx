Anyscale uses a concept of a **cloud** that maps to a Kubernetes cluster with specific
storage, identity, and compute resources registered to it. This module wires the AKS
cluster you deployed in Module 2 to an Anyscale cloud, installs the operator, and
verifies the Gateway API path that Anyscale uses to reach Ray services.

## Objectives

By the end of this module you will be able to:

- Create an Anyscale cloud bound to the AKS cluster
- Install the Anyscale operator via the AKS marketplace extension
- Verify the operator pod and Gateway API resources are healthy
- Confirm the CPU-first compute placement contract before running the proof

## Background: Anyscale on AKS

Anyscale on AKS is available through Azure-native resources and an AKS cluster extension.
The extension installs the Anyscale operator in the `anyscale-operator` namespace and
connects it to the Anyscale control plane at `console.azure.anyscale.com`.

:::note Preview RP deployment uses AzAPI
The Anyscale Azure resource provider path is still preview. Terraform manages it
through AzAPI in this lab because the preview RP shape is not fully represented by
stable first-party Terraform resources yet. Keep the AzAPI path for this workshop.
:::

When a Ray job is submitted, Anyscale places Ray head and worker pods on nodes that
match the compute config's resource requirements. Because the Flex host is a standard
Kubernetes node, Anyscale schedules on it transparently — no workload changes needed.

:::info Preview feature
Anyscale on Azure is currently in public preview. Cross-region scheduling onto Flex hosts
is the scenario this lab explicitly validates. If you encounter unexpected behavior,
capture it in the evidence bundle (Module 6) and open an issue.
:::

## Regional Prerequisites

The Anyscale operator AKS extension is **not available in all regions**. Before deploying,
ensure your AKS cluster is in one of these **supported regions**:

Supported regions can change over time and may differ between subscriptions.
Always preflight extension availability against your exact target AKS cluster.

Use this command before enabling `TF_VAR_anyscale_enabled`:

```bash
az k8s-extension extension-types list-by-cluster \
  --resource-group rg-flexany-dev-wus2 \
  --cluster-name aks-flexany-dev-wus2 \
  --cluster-type managedClusters \
  --query "[?contains(name, 'Microsoft.Anyscale.operator')]"
```

Expected output:

```json
[
  {
    "name": "Microsoft.Anyscale.operator",
    "publisher": "Microsoft.Anyscale"
  }
]
```

If the result is empty, that region/subscription combination does not currently
support the Anyscale operator extension for AKS.

:::caution Regional constraint
The `Microsoft.Anyscale.operator` AKS extension type is **not available in all regions**.
If you attempt to deploy in an unsupported region, you will see:

```text
ExtensionTypeRegistrationGetFailed: Extension type 'Microsoft.Anyscale.operator'
is not supported in region '<region>' for cluster type: ManagedClusters.
```

**Solution**: Deploy your AKS cluster to a region where the preflight command
returns `Microsoft.Anyscale.operator` for your cluster.
:::

## Step 1 — Authenticate to Anyscale

```bash
ANYSCALE_HOST=https://console.azure.anyscale.com \
  .venv/bin/anyscale login
```

Follow the browser prompt. After login, verify:

```bash
.venv/bin/anyscale cloud list
```

## Step 2 — Configure regions and enable Anyscale (if not already set)

**Important**: Module 2 should have configured your regions in `.env`. Verify that your
AKS cluster is in a **supported region** (see "Regional Prerequisites" above).

**Example configuration** (adjust after preflight):

```bash
# In .env:
TF_VAR_azure_location="westus2"       # Home region (AKS cluster location)
TF_VAR_region_short="wus2"
TF_VAR_flex_region="westus3"           # Flex host region
TF_VAR_flex_region_short="wus3"
TF_VAR_anyscale_enabled="true"        # Enable Anyscale operator
```

**Rationale**:

- Home region must support `Microsoft.Anyscale.operator` for your subscription.
- Flex region must have quota for your selected Flex host VM size.
- Cross-region peering demonstrates hybrid AKS + Flex architecture.

If you deployed Module 2 to a different region pair, you must redeploy to a supported region
before continuing with Module 4. See [Module 2 — Deploy the AKS Foundation](/docs/ai-workloads-on-aks/module-02-aks-foundation)
for guidance.

## Step 3 — Deploy the Anyscale operator (Terraform)

The Anyscale cloud resource and the AKS marketplace extension are managed by Terraform
in `infra/terraform/anyscale.tf`. The Anyscale cloud resource uses AzAPI because the
Azure-native Anyscale RP is a preview path.

With `.env` configured correctly (region and `TF_VAR_anyscale_enabled="true"`), deploy:

```bash
./scripts/anyscale-aks.sh apply
```

This creates:

- Azure-native Anyscale cloud resource through AzAPI
- AKS marketplace extension for the Anyscale operator
- Gateway API resources for Anyscale ingress
- Workload Identity federation for secure authentication
- RBAC bindings for operator access

Expected output:

```text
azapi_resource.anyscale_cloud: Creation complete
azurerm_kubernetes_cluster_extension.anyscale_operator: Creation complete
Apply complete! Resources: 4 added, 0 changed, 0 destroyed.
```

Expected duration: **10–15 minutes** (extension reconciliation is slow on first install).

Hard gate: after apply, require Gate 4 to pass before proceeding:

```bash
az k8s-extension show \
  --cluster-name aks-flexany-dev-wus2 \
  --resource-group rg-flexany-dev-wus2 \
  --cluster-type managedClusters \
  --name anyscale-operator \
  --query '{name:name,provisioningState:provisioningState,currentVersion:currentVersion}' -o json
```

Expected output:

```json
{
  "name": "anyscale-operator",
  "provisioningState": "Succeeded",
  "currentVersion": "1.7.1"
}
```

## Step 4 — Verify the operator

```bash
kubectl get pods,gateway -n anyscale-operator -o wide
```

Expected output:

```text
NAME                                     READY   STATUS    RESTARTS   AGE   IP
pod/anyscale-operator-556c4df7cb-tq4r7   3/3     Running   0          8m    10.40.0.25

NAME                                                CLASS              ADDRESS        PROGRAMMED   AGE
gateway.gateway.networking.k8s.io/anyscale-gateway  approuting-istio  20.190.18.76  True         7m
```

Confirm the Gateway has been programmed:

```bash
kubectl get gateway anyscale-gateway \
  -n anyscale-operator \
  -o jsonpath='{.status.conditions}'
```

Expected output:

```json
[{"type":"Programmed","status":"True","reason":"Programmed"}]
```

Confirm the cloud is visible in the console:

```bash
.venv/bin/anyscale cloud list | grep flexany
```

Expected output:

```text
cldrsrc_f3qm4fbbj8gggfp55ewkn5tkeh  flexany-dev-wus2  ACTIVE
```

## Step 5 — Confirm compute placement contract

The organized path uses the repository proof command in Module 6 to submit a CPU-only
Ray job. The expected placement is:

| Step | CPU path | GPU path |
| --- | --- | --- |
| Head pod | AKS CPU node in `westus2` | Future optional GPU-capable AKS node |
| Worker pod | Flex node `agentpool=aksflexnodes` in `westus3` | Future optional GPU Flex node |
| DNS policy | Normal `ClusterFirst` | Normal `ClusterFirst` after GPU enablement |

Do not submit the proof yet. Module 5 performs the final preflight checks, then
Module 6 runs `./scripts/run-anyscale-proof.sh --mode cpu`.

:::tip Optional future GPU configs
GPU compute configs can be added after the CPU proof succeeds and GPU quota is available.
Keep them out of the primary flow so the lab remains repeatable in subscriptions with
no T4 quota.
:::
